# EXECUTION PLAYBOOK

## Purpose

Provide a clear, repeatable workflow for progressing the Prompt OS and product system (DCL and future apps).

---

## Core Rule

Do not rebuild.

Only:
- stabilise
- execute
- test
- refine

---

## Session Start Protocol

1. Review SESSION_STATE.md
2. If needed, use SESSION_RECOVERY_PROMPT.md
3. Identify current phase
4. Execute next step only

---

## Execution Phases

### Phase 1 — System Stabilisation

- confirm single active kernel
- archive redundant files
- ensure clean structure
- separate OS and DCL if needed

---

### Phase 2 — Product Validation (DCL)

- use DCL with real decisions
- identify friction
- improve clarity and speed
- confirm usability

---

### Phase 3 — Deployment

- ensure app is live
- test on multiple devices
- confirm link accessibility
- prepare for sharing

---

### Phase 4 — User Testing

- get 5–10 users
- observe usage
- collect feedback
- identify patterns

---

### Phase 5 — Refinement

- remove friction
- improve outputs
- simplify inputs
- increase clarity

---

### Phase 6 — Launch

- define positioning
- share publicly
- drive traffic
- monitor engagement

---

## Key Principles

- simplicity wins
- speed over perfection
- action over planning
- feedback drives improvement

---

## Warning

Do NOT:
- overbuild
- add unnecessary features
- merge all apps into one
- delay launch

---

## Current Focus

DCL only.

ECL and MCL are future phases.

---

## End Goal

Transition from:
Builder → Operator → Scaler
